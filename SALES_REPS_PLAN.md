## 1. ملخص تنفيذي

### الوضع الحالي

| العنصر | الحالة |
|--------|--------|
| نموذج `SalesRepresentative` في Prisma | موجود ✅ |
| API `/api/sales-reps` GET + POST + [id] | موجود ✅ |
| صفحة `/sales-reps` | **غير موجودة ❌** |
| عزل البيانات (المندوب يرى فواتيره فقط) | موجود ✅ |
| ربط العميل بالمندوب (`Customer.salesRepresentativeId`) | موجود ✅ |
| ربط الفاتورة بالمندوب (`Invoice.salesRepresentativeId`) | موجود ✅ |
| نسبة العمولة ونوعها في الـ schema | موجود ✅ |
| حساب العمولات تلقائياً | **غير موجود ❌** |
| التحصيلات | **غير موجود ❌** |
| أهداف المبيعات | **غير موجود ❌** |

### نموذج `SalesRepresentative` الحالي (ما يوجد في الـ Schema)

```prisma
model SalesRepresentative {
  id             String    @id @default(cuid())
  name           String
  code           String?                          // رمز المندوب
  phone          String?
  email          String?
  commissionRate Float     @default(0)            // نسبة العمولة الافتراضية
  commissionType String    @default("invoice_total")
  // invoice_total = عمولة على إجمالي الفاتورة
  // collected_amount = عمولة على ما تم تحصيله فعلاً
  isActive       Boolean   @default(true)
  userId         String?                          // ربط بمستخدم في النظام
  employeeId     String?                          // ربط بموظف في HR
  companyId      String
  // Relations
  invoices       Invoice[]
  quotations     Quotation[]
  customers      Customer[]
}
```

### ما سيُضاف

- **3 نماذج Prisma جديدة:** Collection, SalesTarget, CommissionPayment
- **5 صفحات جديدة كاملة**
- **تاب تقارير** في صفحة `/reports`
- **قسم مناديب المبيعات** في القائمة الجانبية

## 2. السيكل الكامل

```
┌─────────────────────────────────────────────────────┐
│                    مدير النظام                      │
├─────────────────────────────────────────────────────┤
│ 1. ينشئ مندوب مبيعات (اسم، كود، هاتف، عمولة،     │
│    ربط بحساب مستخدم في النظام)                     │
│ 2. يعيّن عملاء للمندوب                             │
│ 3. يحدد هدف شهري للمندوب                           │
└───────────────────────┬─────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────┐
│               المندوب (حسابه الخاص)                │
├─────────────────────────────────────────────────────┤
│ 4. يدخل النظام ← يرى لوحة تحكم شخصية (KPIs)      │
│ 5. يرى قائمة عملائه فقط + رصيد كل عميل            │
│ 6. ينشئ فاتورة مبيعات                              │
│    ← تدخل بحالة "قيد الاعتماد" (pending)          │
│ 7. يسجّل تحصيل من عميل (نقدي / شيك / تحويل)      │
└───────────────────────┬─────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────┐
│                    مدير النظام                      │
├─────────────────────────────────────────────────────┤
│ 8.  يعتمد الفواتير (approved)                      │
│ 9.  يعتمد التحصيلات ويودعها في الخزينة             │
│ 10. آخر الشهر: يحسب عمولة كل مندوب                │
│ 11. يعتمد العمولات ويصرفها                         │
└─────────────────────────────────────────────────────┘
```

### حالات الفاتورة المرتبطة بالمندوب

| الحالة | من ينشئها | الوصف |
|--------|-----------|-------|
| `pending` قيد الاعتماد | المندوب | تأثير مالي ومخزني معلق |
| `approved` معتمدة | المدير | تأثير مالي ومخزني تام |
| `rejected` مرفوضة | المدير | لا تأثير |

## 3. نماذج قاعدة البيانات الجديدة

### نموذج 1: `Collection` — التحصيلات

```prisma
model Collection {
  id              String    @id @default(cuid())
  date            DateTime
  amount          Float
  method          String    // cash | check | transfer
  status          String    @default("pending")
  // pending   = معلق (لم يُودع بعد)
  // deposited = تم إيداعه في الخزينة
  // returned  = مرتجع (شيك مرتجع أو رفض)

  // من ومن
  salesRepId      String
  customerId      String
  invoiceId       String?   // الفاتورة المحصّلة (اختياري)

  // بيانات الشيك (إن كانت الطريقة شيك)
  checkNumber     String?
  bankName        String?
  checkDueDate    DateTime?

  // الإيداع
  treasuryId      String?   // الخزينة التي أُودع فيها
  depositedAt     DateTime?
  depositedBy     String?   // userId

  notes           String?
  companyId       String
  branchId        String?
  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt

  salesRep        SalesRepresentative @relation(fields: [salesRepId], references: [id])
  customer        Customer            @relation(fields: [customerId], references: [id])
  invoice         Invoice?            @relation(fields: [invoiceId], references: [id])
  treasury        Treasury?           @relation(fields: [treasuryId], references: [id])
  company         Company             @relation(fields: [companyId], references: [id])

  @@index([companyId, date])
  @@index([salesRepId, status])
  @@index([customerId])
}
```

### نموذج 2: `SalesTarget` — الأهداف الشهرية

```prisma
model SalesTarget {
  id             String    @id @default(cuid())
  salesRepId     String
  year           Int
  month          Int       // 1-12
  targetAmount   Float     // هدف المبيعات بالمبلغ
  targetCount    Int?      // هدف عدد الفواتير (اختياري)
  notes          String?
  companyId      String
  createdAt      DateTime  @default(now())
  updatedAt      DateTime  @updatedAt

  salesRep       SalesRepresentative @relation(fields: [salesRepId], references: [id])
  company        Company             @relation(fields: [companyId], references: [id])

  @@unique([salesRepId, year, month])
  @@index([companyId, year, month])
}
```

### نموذج 3: `CommissionPayment` — سجل صرف العمولات

```prisma
model CommissionPayment {
  id               String    @id @default(cuid())
  salesRepId       String
  year             Int
  month            Int

  // الأرقام الأساسية للحساب
  totalSales       Float     // إجمالي مبيعات الشهر (مبلغ الفواتير المعتمدة)
  totalCollected   Float     // إجمالي ما تم تحصيله فعلاً
  commissionBase   Float     // الأساس المستخدم (depends on commissionType)
  commissionRate   Float     // النسبة المطبقة
  commissionAmount Float     // العمولة المستحقة = commissionBase × commissionRate

  // حالة الصرف
  status           String    @default("calculated")
  // calculated = محسوبة فقط
  // approved   = معتمدة من المدير
  // paid       = تم الصرف

  approvedBy       String?   // userId
  approvedAt       DateTime?
  paidBy           String?   // userId
  paidAt           DateTime?
  treasuryId       String?   // الخزينة التي صُرفت منها

  notes            String?
  companyId        String
  createdAt        DateTime  @default(now())
  updatedAt        DateTime  @updatedAt

  salesRep         SalesRepresentative @relation(fields: [salesRepId], references: [id])
  treasury         Treasury?           @relation(fields: [treasuryId], references: [id])
  company          Company             @relation(fields: [companyId], references: [id])

  @@unique([salesRepId, year, month])
  @@index([companyId, year, month])
  @@index([status])
}
```

## 7. صفحة العمولات

**المسار:** `/sales-reps/commissions`  
**الحالة:** جديدة كلياً  
**الصلاحية:** مدير فقط

### الفلاتر

```
[الشهر: يونيو 2026 ▼]   [المندوب: الكل ▼]   [الحالة: الكل ▼]
                    [احسب العمولات للشهر]
```

### الجدول

| المندوب | إجمالي المبيعات | تم تحصيله | أساس الحساب | النسبة | العمولة | الحالة | إجراء |
|---------|----------------|-----------|-------------|--------|---------|--------|-------|
| أحمد محمد | 48,200 | 40,000 | 48,200 | 3% | 1,446 | 🔵 محسوبة | اعتماد |
| محمد علي | 72,000 | 72,000 | 72,000 | 2.5% | 1,800 | 🟡 معتمدة | صرف |
| سارة خالد | 31,500 | 25,000 | 25,000 | 4% | 1,000 | 🟢 مدفوعة | — |

**ملاحظة:** "أساس الحساب" يعتمد على `commissionType` لكل مندوب:
- `invoice_total` → أساس الحساب = إجمالي المبيعات
- `collected_amount` → أساس الحساب = ما تم تحصيله

### آلية حساب العمولات

```
زر "احسب العمولات" ← API يقوم بـ:

1. يجلب كل الفواتير المعتمدة (status='approved') للشهر لكل مندوب
   totalSales = sum(invoices.total)

2. يجلب كل التحصيلات المودَعة (status='deposited') للشهر لكل مندوب
   totalCollected = sum(collections.amount)

3. commissionBase = (commissionType === 'invoice_total') ? totalSales : totalCollected

4. commissionAmount = commissionBase × (commissionRate / 100)

5. ينشئ/يحدّث CommissionPayment للشهر
```

### سيكل اعتماد وصرف العمولة

```
محسوبة (calculated)
    ↓ [مدير يضغط "اعتماد"]
معتمدة (approved)
    ↓ [مدير يضغط "صرف" ويختار الخزينة]
مدفوعة (paid) ← يُخصم من الخزينة ويُسجَّل في سجل النشاط
```

### سجل المدفوعات السابقة

جدول يعرض كل عمولات المندوب المدفوعة تاريخياً:

| الشهر | إجمالي المبيعات | العمولة | تاريخ الصرف | المعتمد من |
|-------|----------------|---------|-------------|-----------|
| مايو 2026 | 45,000 | 1,350 | 2026-06-05 | مدير النظام |
| أبريل 2026 | 38,200 | 1,146 | 2026-05-03 | مدير النظام |

## 12. مخطط قاعدة البيانات

### النماذج الجديدة المطلوب إضافتها

```
النظام الحالي (موجود):
├── SalesRepresentative (موجود ✅)
│   ├── مرتبط بـ Invoice   ✅
│   ├── مرتبط بـ Quotation  ✅
│   └── مرتبط بـ Customer   ✅

النماذج الجديدة:
├── 🆕 Collection          ← تحصيلات المناديب
├── 🆕 SalesTarget         ← الأهداف الشهرية
└── 🆕 CommissionPayment   ← سجل صرف العمولات

التعديلات على النماذج الموجودة:
├── 📝 SalesRepresentative: + collections[] + targets[] + commissions[]
├── 📝 Customer:            + collections[]
└── 📝 Company:             + collections[] + salesTargets[] + commissionPayments[]
```

### APIs المطلوب إنشاؤها

```
/api/collections
    GET  — قائمة التحصيلات (فلاتر: salesRepId, status, method, dateFrom, dateTo)
    POST — تسجيل تحصيل جديد

/api/collections/[id]
    GET    — تفاصيل تحصيل
    PATCH  — تحديث (إيداع أو إرجاع)
    DELETE — حذف (معلق فقط)

/api/sales-targets
    GET  — قائمة الأهداف (فلاتر: year, month, salesRepId)
    POST — إنشاء / تحديث هدف

/api/sales-targets/[id]
    PUT    — تعديل هدف
    DELETE — حذف هدف

/api/commissions
    GET  — قائمة العمولات (فلاتر: year, month, salesRepId, status)
    POST — حساب عمولات شهر معين (يُنشئ CommissionPayment)

/api/commissions/[id]
    PATCH — اعتماد أو صرف عمولة
```

## ملاحظات مهمة

1. **عزل البيانات للمندوب:** المستخدم المرتبط بـ `SalesRepresentative` يرى فقط:
   - فواتيره الخاصة (موجود ✅)
   - تحصيلاته الخاصة (يُضاف مع Collections)
   - عمولاته الخاصة (للعرض فقط — لا يعتمد أو يصرف)
   
2. **العمولة على التحصيل:** إذا كان `commissionType = 'collected_amount'`، فالعمولة تُحسب على مجموع `Collection.amount` المودَعة للشهر وليس على إجمالي الفواتير

3. **الشيكات المرتجعة:** عند تغيير حالة تحصيل من `deposited` إلى `returned`، يُخصم المبلغ من الخزينة مرة أخرى (reverse transaction)

4. **الصلاحيات:** يُضاف قسم مناديب المبيعات لنظام الصلاحيات الموجود حتى يمكن التحكم في من يرى/يعدّل كل صفحة

5. **التكامل مع الرواتب:** عند ربط `CommissionPayment` بـ `employeeId`، يمكن في المستقبل سحب العمولات تلقائياً في مسير الرواتب
