# سيكل نشاط الخدمات - تحليل شامل

---

## أولاً: السيكل الكامل المفروض

```
عميل
  ↓
عقد خدمة (اختياري)
  ↓
أمر عمل
  ↓
تعيين فني مسؤول
  ↓
بدء التنفيذ (تسجيل وقت البداية)
  ↓
إضافة المواد المستخدمة من المخزن
  ↓
كتابة ما تم عمله (Resolution)
  ↓
إتمام أمر العمل (تسجيل وقت الانتهاء)
  ↓
إنشاء فاتورة للعميل
  ↓
تحصيل المدفوعات
```

## رابعاً: مسارات API الموجودة

### عقود الخدمة

| الطريقة | المسار | الوظيفة |
|---------|--------|---------|
| GET | `/api/service-contracts` | جلب كل العقود مع فلترة (الحالة، العميل، النوع) |
| POST | `/api/service-contracts` | إنشاء عقد جديد بحالة draft تلقائياً |
| GET | `/api/service-contracts/[id]` | جلب تفاصيل عقد مع العميل وأوامر العمل |
| PUT | `/api/service-contracts/[id]` | تعديل بيانات العقد أو تغيير الحالة |
| DELETE | `/api/service-contracts/[id]` | حذف (فقط إذا كان بحالة draft) |

| الطريقة | المسار | الوظيفة |
|---------|--------|---------|

| POST | `/api/work-orders` | إنشاء أمر جديد مع مواد اختيارية |

## خامساً: سيكل الحالات (Status Flow)

### عقد الخدمة
```
draft (مسودة)
    ↓ تفعيل يدوي
active (نشط)
    ↓                    ↓
expired (منتهي)    cancelled (ملغي)
    ↓
active (إعادة تفعيل يدوية)
```

### أمر العمل
```
new (جديد) ──────→ assigned (مُسنَد) ──→ in_progress (قيد التنفيذ) ──→ completed (مكتمل)
   ↓                    ↓                        ↓
   └────────────────────┴────────────────────────┴──→ cancelled (ملغي)
```

**ملاحظات الحالات:**
- `new → assigned`: تلقائي عند تعيين فني في الفورم
- `new/assigned → in_progress`: يدوي، يُسجل `startedAt` تلقائياً
- `in_progress → completed`: يدوي، يُسجل `completedAt` تلقائياً
- `completed/cancelled`: لا يمكن تغيير الحالة بعدهم

## سادساً: السيكل الكامل خطوة بخطوة (الحالي)

### المسار الأول: مع عقد خدمة

```
1. إنشاء عقد خدمة
   ← اختيار العميل
   ← تحديد نوع الخدمة (صيانة / استشارات / تطوير / دعم فني)
   ← تحديد مدة العقد وقيمته ودورة الفوترة
   ← العقد يُحفظ بحالة "مسودة"

2. تفعيل العقد
   ← من صفحة تفاصيل العقد
   ← الضغط على "تفعيل العقد"
   ← يصبح الحالة "نشط"

3. إنشاء أمر عمل من العقد
   ← من صفحة العقد → "إنشاء أمر عمل"
   ← تُملأ بيانات العميل والعقد تلقائياً
   ← تحديد النوع والأولوية والفني والتاريخ المجدول
   ← الأمر يُحفظ بحالة "جديد" أو "مُسنَد"

4. بدء التنفيذ
   ← من تفاصيل أمر العمل
   ← الضغط على "بدء التنفيذ"
   ← يُسجل وقت البداية تلقائياً
   ← الحالة تصبح "قيد التنفيذ"

5. إتمام أمر العمل
   ← الضغط على "إتمام أمر العمل"
   ← يُسجل وقت الانتهاء تلقائياً
   ← الحالة تصبح "مكتمل"
```

### المسار الثاني: بدون عقد (أمر عمل مباشر)

```
1. إنشاء أمر عمل مباشرة
   ← بدون ربط بعقد
   ← اختيار عميل مباشرة
   ← نفس الخطوات من 3 إلى 5 في المسار الأول
```

## سابعاً: ما هو موجود في قاعدة البيانات لكن غير مكتمل في الواجهة

| الميزة | موجود في DB | موجود في API | موجود في الواجهة |
|--------|-------------|-------------|-----------------|
| بيانات العقد الأساسية | ✅ | ✅ | ✅ |
| أوامر العمل الأساسية | ✅ | ✅ | ✅ |
| المواد المستخدمة | ✅ | ✅ | ❌ (تُعرض فقط، لا تُضاف) |
| حقل "ما تم عمله" (Resolution) | ✅ | ✅ | ❌ (يُعرض فقط، لا يُكتب) |
| تحويل أمر عمل لفاتورة | ✅ (علاقة موجودة) | ❌ | ❌ |
| تعديل أمر العمل | ✅ | ✅ | ❌ (الصفحة مش موجودة) |

## ثامناً: المشاكل الحرجة المطلوب حلها

### 1. صفحة تعديل أمر العمل مش موجودة ❌ (الأكثر أهمية)

- **المشكلة:** زر "تعديل أمر العمل" موجود في صفحة التفاصيل ويحاول يفتح `/work-orders/[id]/edit` لكن الصفحة لا وجود لها.
- **التأثير:** أي خطأ في بيانات أمر العمل لا يمكن تصحيحه من الواجهة.
- **الحل:** إنشاء صفحة تعديل كاملة تسمح بتعديل البيانات وإضافة المواد وكتابة ما تم عمله، وتكون متاحة فقط إذا كانت الحالة `new` أو `assigned`.

---

### 2. لا يوجد UI لإضافة المواد المستخدمة ❌

- **المشكلة:** الـ API يدعم المواد بشكل كامل، وجدول المواد يظهر في صفحة التفاصيل، لكن لا يوجد فورم لإضافتها أو تعديلها.
- **التأثير:** لا يمكن تسجيل المواد المستهلكة من المخزن لإتمام العمل، مما يعني عدم دقة المخزون وعدم حساب التكلفة الحقيقية.
- **الحل:** إضافة جدول تفاعلي لإضافة وحذف المواد في فورم الإنشاء والتعديل.

---

### 3. لا يوجد حقل "ما تم عمله" في الفورم ❌

- **المشكلة:** حقل `resolution` موجود في الـ database ويظهر في التفاصيل بخلفية خضراء، لكن لا يوجد طريقة لملأه من الواجهة.
- **التأثير:** لا يمكن توثيق ما قام به الفني فعلياً.
- **الحل:** إضافة حقل نصي لكتابة ما تم عمله عند الضغط على "إتمام أمر العمل" أو في صفحة التعديل.

---

### 4. لا يوجد تحويل أمر عمل مكتمل لفاتورة ❌

- **المشكلة:** السيكل يقف عند "مكتمل" ولا يوجد زر "إنشاء فاتورة" لتحصيل قيمة الخدمة من العميل.
- **التأثير:** الحلقة مكسورة - الشركة تنفذ العمل لكن لا تستطيع إنشاء الفاتورة منه مباشرة.
- **الحل:** إضافة زر "إنشاء فاتورة" في صفحة تفاصيل أمر العمل المكتمل، ينقل بيانات العميل والمواد والقيمة تلقائياً لصفحة إنشاء الفاتورة.

## تاسعاً: ميزات مبنية في DB لكن غير مُفعَّلة

| الميزة | الوضع | الملاحظة |
|--------|-------|---------|
| التجديد التلقائي (autoRenew) | موجود في DB فقط | لا يوجد جدولة تجدد العقد تلقائياً عند انتهاء تاريخه |
| دورة الفوترة (billingCycle) | موجود في DB فقط | لا تُنشأ فواتير دورية تلقائياً بناءً على دورة الفوترة |
| الفرع (branchId) | موجود في DB فقط | لا يُستخدم في الفلترة ولا في الواجهة |
| صلاحيات تغيير الحالة | غير موجودة | أي مستخدم يمكنه تغيير حالة أي أمر عمل |
